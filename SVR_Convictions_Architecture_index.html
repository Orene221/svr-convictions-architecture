Trois principes avant toute conviction

Le point de départ est votre stack, pas une plateforme cible. Voici comment nous avons structuré l'analyse.

01
L'infrastructure suit les usages — pas l'inverse

Aucune recommandation plateforme ne précède l'analyse des cas d'usage. Ce document répond UC par UC : qu'est-ce qui marche sur votre stack actuelle, et qu'est-ce qui bloque.

02
Votre socle Oracle est une force, pas un problème

Oracle DWH, Sage X3, Oracle AI DB 26ai, TTMD — tout cela fonctionne et représente un actif. Nous ne recommandons aucune migration de ce qui marche. Le POC TTMD en cours le confirme.

03
La couche agentique est la seule zone de tension réelle

Pour les UC T1 et la plupart des UC T2, votre infrastructure suffit. La tension apparaît uniquement sur les ambitions T3 — agents autonomes, AI Factory, surveillance proactive — qui nécessitent un framework d'orchestration qu'Oracle ne fournit pas nativement.

Notre conviction architecturale

Bi-cloud complémentaire — pas un choix, une logique

La recommandation n'est pas "quittez OCI pour Azure". C'est : chaque plateforme couvre ce pour quoi elle est faite, sans migration inutile. C'est également la conclusion de notre benchmark cloud ETI (mai 2026) : "Architecture bi-cloud ciblée plutôt qu'un choix unique."

Pourquoi cette architecture, et pas "tout OCI" ou "tout Azure" ?

  • Migrer votre Oracle DWH vers Azure serait une dépense sans valeur : vos données Oracle fonctionnent dans Oracle. Vos rapports BI aussi. On ne touche pas à ce qui fonctionne.
  • Rester "tout OCI" pour T3 crée un mur technique sur l'orchestration agentique — ce n'est pas une limite temporaire, c'est un choix de positionnement d'Oracle. OCI investit sur les données, pas sur les agents.
  • Azure n'est pas "mieux" qu'OCI. Il est complémentaire sur la couche agentique et M365 — la seule couche que vous ne pouvez pas adresser avec votre stack actuelle.
  • La connexion Oracle ↔ Azure existe via connecteurs certifiés : vos données Oracle alimenteront les agents Azure sans ETL coûteux. C'est la conclusion de notre benchmark ETI mai 2026.
  • Sur Power BI vs Oracle BI : Oracle BI reste le bon outil pour vos rapports ERP. Power BI apporte une valeur si vous avez déjà des licences M365 E5 et souhaitez des requêtes en langage naturel sur la BI — à évaluer avec Christophe Servolin, pas une priorité immédiate.

Ce que nous vous demandons de valider

Deux décisions, pas une

Nous ne vous demandons pas de valider Azure vs OCI. Nous vous demandons de valider l'ambition T3 et le timing de l'étude qui permet de trancher.

Décision 1 — Ambition stratégique
L'AI Factory T3 — agents autonomes SC, NBO proactif, extension groupe — est-elle votre cible pour 2028+ ?
Oui → lancer l'étude archi T4 2026 (70-120 k€) pour concevoir l'architecture qui permet T3 sans refonte · AI Engineer recruté Q4 2026 avec stack définie
Pas encore → concentrer T1/T2 sur OCI et M365 · reporter l'arbitrage archi à T2 · risque de réorientation à la charge des prestataires
Décision 2 — Timing & pilotage étude
Quand est lancée l'étude archi, et qui la pilote côté SVR/KRESK ?
T4 2026 (recommandé) → Andry (DSI) + CTO (arrivé juin) · Wavestone + experts OCI (Yves Rombert / Roland Caron) · avant recrutement AI Engineer
T2 2027 → après bilan T1 · risque : AI Engineer recruté Q4 2026 sans architecture cible · prestataires T2 buildent sur OCI par défaut
Notre position, sans ambiguïté : Les 8 UC T1 fonctionnent sur votre infrastructure actuelle — aucune décision cloud n'est requise avant mi-juin 2026. Pour T2, OCI couvre VE2 ML et RD2 exploration ; Azure/M365 couvre FI1, SC1, GP, MK1 via les licences existantes. La seule tension structurelle est sur l'orchestration agentique T3. L'étude de 70-120 k€ n'est pas un pari sur une plateforme : c'est l'achat de la certitude sur ce que vous buildez en T3 — et la condition pour recruter un AI Engineer avec un brief technique solide.
